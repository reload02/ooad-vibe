# 🤖 UP 기반 AI 협업 및 개발 프로세스 지침서 (Project: Gateway)

본 지침서는 Unified Process(UP)의 4대 단계(Inception, Elaboration, Construction, Transition)와 Dev Iteration 및 Ops 환경에서 AI Agent와 인간 개발자가 협업하는 규칙을 정의합니다.

---

## 💡 기본 원칙

1. **책임의 주체**: 프로젝트의 존폐를 결정짓거나 법적·비용적 리스크가 있는 핵심 결정 사항은 반드시 **인간이 주체**가 되어 검증합니다.
2. **환경 선구축**: AI에게 작업을 지시하기 전, AI가 올바르게 작동할 수 있는 프롬프트, 하네스, CI/CD 등 **기반 환경을 먼저 구축**해야 합니다.
3. **반복(Iteration) 지향**: 개발은 반복 주기(Iteration) 단위로 진행되며, 매 Iteration 시작 전 수행할 영역을 스크럼을 통해 결정합니다. 한 Iteration 내에서 모든 단계를 다 거칠 필요는 없으며, **현재 프로젝트의 단계에 맞는 산출물(Artifact)을 도출**하는 데 집중합니다.

---

## 1. 단계별(Phase) 프로세스 및 산출물 가이드

### ① 도입 단계 (Inception)

프로젝트의 타당성을 검토하고 요구사항의 기초를 다지는 단계입니다. 잘못된 요구사항 파악은 전체 개발 방향을 흔들 수 있으므로 인간의 엄격한 검증이 필요합니다.

- **진행 방식**: `Human Only` 주도 후 AI 보조.

- **수행 활동**:
- **Feasibility Study**: 리스크 분석 및 비용 예측 (비용 연관으로 인간의 책임).

- **CI/CD Design**: 인프라 비용 및 아키텍처 설계 (인간의 책임).

- **Requirement Workshop**: 핵심 요구사항 정의 및 Actor 파악.

- **필수 산출물**:
  - `Use Case Model` (정확한 Actor 및 요구사항 명시 문서)

### ② 구체화 단계 (Elaboration)

시스템의 아키텍처를 설계하고, AI가 고품질의 결과물을 일관되게 뱉어낼 수 있도록 '하네스(Harness)'를 구축하는 단계입니다.

- **진행 방식**: `Human + AI` 협업.

- **수행 활동**:
- **Harness Prompting / Document**: AI용 컨텍스트 및 규칙 설계.

- **Diagramming**: AI의 도움을 받아 설계 다이어그램을 작성하고 알고리즘/로직 검증.

- **Prototype**: AI를 통해 빠르게 프로토타입을 빌드하여 핵심 가치 검증 및 테스트 케이스 발굴.

- **필수 산출물**:
  - `Harness Document` (AI 지시용 가이드라인)
  - `Domain Diagram & System Sequence Diagram`
  - `Class Diagram & Sequence Diagram`
  - `SRS (사용자 요구사항 명세서) & SDD (소프트웨어 설계 설명서)` (이해관계자 공유 표준 문서)
  - `Test Cases Scenario` (인간이 설계한 테스트 시나리오 구조)

### ③ 구축 단계 (Construction)

실제 시스템 코드와 테스트 코드를 완성하는 단계입니다.

- **진행 방식**: PLAN 모드를 기반으로 한 AI 중심 개발 및 인간의 최종 승인.

- **수행 활동**:
- **System Code 작성**: 설계 문서를 기반으로 AI가 구현.

- **Test Code 작성**: Elaboration에서 설계한 시나리오를 바탕으로 AI가 테스트 코드 작성 및 수행.

- **필수 산출물**:
  - `System Code` (인간의 검수를 거쳐 최종 승인된 소스코드)
  - `Test Code` (유효성 검증이 완료된 테스트 스크립트)
  - `AI Context file` (코드 변경점 및 구조를 풀어서 표현한 AI용 보조 문서)

### ④ 전이 단계 (Transition)

제품을 사용자에게 배포하고 유지보수 기반을 마련하는 단계입니다.

- **진행 방식**: CI/CD 자동화 환경 및 인간의 최종 릴리즈 승인.

- **필수 산출물**:
  - `Release Note` (최종 빌드 기능 목록, 개선 사항, 버그, 제약 사항 명시)
  - `Development Guideline` (팀 및 오픈소스 기여자를 위한 개발 방식 가이드)
  - `Final Product` (정식 배포 버전 제품)

## 2. 개발 주기(Dev Iteration) 중 AI 활용 지침

매 Iteration은 프로젝트의 현재 단계에 맞춰 유연하게 운영되나, **코드 작성 및 수정 주기**에서는 다음 프로세스를 엄격히 준수합니다.

```
[Scrum (계획)] ➔ [Harness Prompting] ➔ [Make Code] ➔ [Pass Test (TDD)] ➔ [Pair Code Analysis (검증)]

```

1.  **계획 수립 (Scrum)**: Iteration 시작 전, 이번 주기에서 완수할 목표와 작성할 산출물을 스크럼을 통해 결정합니다.

2.  **구현 시 PLAN 모드 활용**: AI에게 처음부터 코드를 짜게 하지 말고, **PLAN 모드로 먼저 구현 계획을 수립하게 한 뒤** 진행하십시오. 결과물의 품질이 훨씬 뛰어납니다.

3.  **명확한 원칙 제시 (SOLID)**: 코드를 요청할 때는 설계 원칙(예: SOLID)을 프롬프트에 명확히 제시해야 합니다. AI가 특정 컴포넌트(예: Simulator)에만 원칙을 적용하고 내부 로직은 절차지향적으로 짜는 경향이 있으므로 컴포넌트 간, 그리고 내부 구조 모두에 원칙이 적용되는지 확인하십시오.

4.  **테스트 주도 개발 (TDD) 및 인간 검증**:
    - AI는 짧은 시간 안에 많은 테스트 코드를 만들어내지만, 무의미하거나 쓸데없는 테스트가 포함될 수 있으므로 **실제 유용한 테스트인지 인간이 검증**해야 합니다.
    - **주의**: AI는 테스트 실패 시 코드를 고치는 게 아니라, *테스트가 통과하도록 테스트 자체를 왜곡*하는 경향이 있습니다. 테스트 불통 시 인간이 직접 코드를 고치거나 가이드를 주어 TDD를 수행하십시오.

5.  **교차 코드 분석 (Pair Code Analysis)**: AI가 작성한 직관적이지 않은 코드나 한 줄에 모든 로직을 구겨 넣으려는 경향(`축약 버그`)을 방지하기 위해, 인간과 AI가 교차 검증을 수행하고 주석 및 보조 문서를 보강합니다.

---

## 3. 운영 단계 (Ops) 중 AI 활용 지침

제품 배포 후 운영(Ops) 및 유지보수 단계에서의 역할 분담 정의입니다.

- **인간 전용 영역 (Human Only)**:

- **Cost Management**: 비용 관리 및 스토리징 전략 수립, 불필요한 인스턴스 정리.

- **Infrastructure Improvements**: 고가용성, 장애 대비, 접속 시간 향상 등 인프라 아키텍처 개선.

- **인간 + AI 협업 영역 (Human + AI)**:

- **Logging & Log 분석**: AI를 활용해 로그를 수집·분석하여 프로젝트의 문제점 및 개선점을 파악합니다. AI의 분석 리포트를 바탕으로 **인간이 판단하여 리팩토링 여부를 결정**합니다.

- **Refactoring**: 코드 품질 검증 및 구조 개선.

- **Add Features (기능 추가)**: 크지 않은 부가 기능 위주로 추가하되, **큰 규모의 기능 변경이 발생할 경우 Ops 단계에서 처리하지 않고 다시 Inception(도입) 단계로 돌아가 검증부터 선행**합니다.

## 🚨 개발자 경고: '바이브 코딩(Vibe Coding)' 경계령

AI에게 모든 것을 일임하고 결과물에 컨펌만 하는 '바이브 코딩'은 단기적으로 빨라 보이지만 프로젝트를 파멸로 이끌 수 있습니다.

- **문제점**: 요구사항 변경 시 AI가 여러 군데를 임의로 수정하여 인간이 변경점을 파악하기 힘들어지며, 인터럽트 처리 누락이나 잘못된 명명법(예: `FrontObstacleInterrupt`가 인터럽트로 동작하지 않는 현상) 등의 크리티컬한 버그를 놓치기 쉽습니다.

- **해결책**: 조금 느리더라도 조원 모두가 구조를 명확히 인지하고 조율하는 **오라클(Oracle) 방식**의 아키텍처 중심 개발을 유지하십시오. AI가 짠 코드는 무조건 **쉽게 풀어 표현한 문서(`AI Context file`)나 보조 도구를 곁들여 직관성을 확보**해야 합니다.
